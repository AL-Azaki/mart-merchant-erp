# المواصفات الوظيفية المؤسسية لمحرك سير العمل والموافقات
**المشروع:** Smart Merchant ERP  
**النطاق:** Workflow & Approval Engine Module  
**الإصدار:** 2.0 (النسخة المحسّنة والموحّدة)  
**الهدف:** توثيق شامل ومتكامل لكافة متطلبات، عمليات، سياسات، وقيود محرك سير العمل والموافقات، لتكون مرجعاً هندسياً وتشغيلياً مغلقاً للمطورين ومحللي الأعمال.

---

## 1. الأهداف الإستراتيجية والنطاق

### 1.1 الأهداف
- توفير محرك مرن وقابل للتخصيص لإدارة سير العمل والموافقات في جميع وحدات النظام.
- أتمتة العمليات التجارية وتقليل الاعتماد على التدخل اليدوي.
- ضمان الامتثال للسياسات والإجراءات من خلال تطبيق قواعد الموافقة بشكل متسق.
- توفير رؤية كاملة لحالة المعاملات (قيد الموافقة، معتمدة، مرفوضة).
- تحسين الكفاءة التشغيلية من خلال تقليل وقت المعالجة والتأخير.

### 1.2 النطاق
يشمل النطاق إدارة تعريفات سير العمل (Workflow Definitions)، إدارة قواعد الموافقة (Approval Rules)، إدارة المهام (Tasks)، إدارة الإشعارات المرتبطة بسير العمل، تتبع حالة المعاملات، وإدارة سجل التدقيق الكامل لجميع عمليات الموافقة.

### 1.3 المسؤوليات الأساسية
- **التصميم:** تصميم سير العمل وقواعد الموافقة بما يتوافق مع سياسات الشركة.
- **التنفيذ:** تنفيذ سير العمل بشكل آلي ودقيق.
- **المتابعة:** متابعة المعاملات المعلقة وضمان معالجتها في الوقت المناسب.
- **التوثيق:** توثيق جميع عمليات الموافقة وسير العمل.
- **التحسين:** تحسين سير العمل بشكل مستمر بناءً على التغذية الراجعة والتحليلات.

## 2. الأمان والصلاحيات (الحوكمة)

### 2.1 أدوار المستخدمين (User Roles)
| الدور | المسؤولية الأساسية |
| :--- | :--- |
| **مستخدم (User)** | تقديم الطلبات، متابعة حالة طلباته. |
| **مدقق (Approver)** | مراجعة واعتماد أو رفض الطلبات الموكلة إليه. |
| **مدير سير العمل (Workflow Manager)** | تصميم وتعديل سير العمل وقواعد الموافقة. |
| **مدير النظام (System Admin)** | إدارة صلاحيات المستخدمين، مراقبة أداء المحرك. |
| **مدقق داخلي (Internal Auditor)** | مراجعة سجلات الموافقات، التأكد من الامتثال. |

### 2.2 الصلاحيات الأساسية (Permissions)
- `Create_Workflow` (إنشاء تعريف سير عمل جديد)
- `Edit_Workflow` (تعديل تعريف سير عمل)
- `Delete_Workflow` (حذف تعريف سير عمل - مقيد جداً)
- `Activate_Workflow` (تفعيل سير عمل)
- `Deactivate_Workflow` (إلغاء تفعيل سير عمل)
- `Manage_Approval_Rules` (إدارة قواعد الموافقة)
- `Approve_Request` (اعتماد طلب)
- `Reject_Request` (رفض طلب)
- `Delegate_Approval` (تفويض الموافقة لمستخدم آخر)
- `View_Workflow_Status` (عرض حالة سير العمل)
- `View_Audit_Log` (عرض سجل التدقيق)
- `Override_Approval` (تجاوز الموافقة - صلاحية مقيدة جداً)

### 2.3 القواعد الأمنية (Security Rules)
- **فصل المهام:** يمنع أن يكون نفس الشخص هو من يقدم الطلب ويوافق عليه.
- **الموافقة المزدوجة:** تتطلب المعاملات الكبيرة أو الحساسة موافقتين أو أكثر.
- **التفويض المحدود:** يمكن تفويض صلاحية الموافقة فقط للمستخدمين المصرح لهم.
- **سجل التدقيق:** تسجيل كل خطوة في سير العمل (من أنشأ، متى، ماذا، ومن وافق/رفض).
- **صلاحية التجاوز:** صلاحية التجاوز تكون محدودة جداً لكبار المديرين فقط.

## 3. تعريفات سير العمل (Workflow Definitions)

### 3.1 هيكل تعريف سير العمل
| الحقل | الوصف | إلزامي |
| :--- | :--- | :--- |
| **الاسم (Name)** | اسم تعريف سير العمل. | نعم |
| **الرمز (Code)** | رمز فريد لتعريف سير العمل. | نعم |
| **الوصف (Description)** | وصف مختصر لسير العمل. | لا |
| **الوحدة (Module)** | الوحدة المرتبطة (مبيعات، مشتريات، موارد بشرية). | نعم |
| **نوع المعاملة (Transaction Type)** | نوع المعاملة (فاتورة، أمر شراء، طلب إجازة). | نعم |
| **الحالة (Status)** | مسودة، نشط، غير نشط، مؤرشف. | نعم |
| **الإصدار (Version)** | رقم إصدار تعريف سير العمل. | نعم |
| **تاريخ الإنشاء (Created Date)** | تاريخ إنشاء التعريف. | نعم |
| **تاريخ التعديل (Modified Date)** | تاريخ آخر تعديل. | لا |

### 3.2 مكونات سير العمل
| المكون | الوصف |
| :--- | :--- |
| **الخطوات (Steps)** | تسلسل الخطوات التي تمر بها المعاملة. |
| **الشروط (Conditions)** | الشروط التي تحدد مسار سير العمل. |
| **القواعد (Rules)** | قواعد الموافقة والتحويل والإشعارات. |
| **الأدوار (Roles)** | الأدوار المسؤولة عن كل خطوة. |
| **المستخدمون (Users)** | مستخدمون محددون مسؤولون عن خطوات معينة. |
| **الإشعارات (Notifications)** | الإشعارات التي يتم إرسالها في كل خطوة. |
| **المهلة (Timeout)** | المهلة الزمنية لكل خطوة. |

### 3.3 أمثلة على تعريفات سير العمل
**سير عمل فاتورة المبيعات:**
1.  إنشاء الفاتورة (المستخدم).
2.  مراجعة الفاتورة (مدير المبيعات).
3.  موافقة ائتمانية (مدير المالية) - إذا تجاوزت الفاتورة حداً معيناً.
4.  موافقة نهائية (المدير التنفيذي) - إذا تجاوزت الفاتورة حداً أعلى.
5.  ترحيل الفاتورة (تلقائي).

**سير عمل طلب الإجازة:**
1.  تقديم طلب الإجازة (الموظف).
2.  مراجعة الطلب (المشرف المباشر).
3.  موافقة الموارد البشرية (أخصائي الموارد البشرية) - للإجازات الطويلة.
4.  إشعار الموظف بالنتيجة (تلقائي).

## 4. قواعد الموافقة (Approval Rules)

### 4.1 أنواع قواعد الموافقة
| النوع | الوصف |
| :--- | :--- |
| **موافقة تسلسلية (Sequential)** | الموافقات تتم بشكل متسلسل (واحد تلو الآخر). |
| **موافقة متوازية (Parallel)** | الموافقات تتم بشكل متوازي (جميع المدققين في نفس الوقت). |
| **موافقة هرمية (Hierarchical)** | الموافقات حسب الهرم الوظيفي (المشرف، مدير القسم، المدير العام). |
| **موافقة شرطية (Conditional)** | الموافقات تعتمد على شروط محددة (مثل قيمة المعاملة). |
| **موافقة دائرية (Round-Robin)** | توزيع طلبات الموافقة بشكل دوري على مجموعة من المدققين. |
| **موافقة الأغلبية (Majority)** | تحتاج موافقة أغلبية المدققين (للجان). |

### 4.2 شروط الموافقة
| الشرط | الوصف | مثال |
| :--- | :--- | :--- |
| **قيمة المعاملة (Amount)** | بناءً على المبلغ المالي. | > 10,000 ريال يحتاج موافقة المدير المالي. |
| **نوع المعاملة (Type)** | بناءً على نوع المعاملة. | فواتير المبيعات تحتاج موافقة مدير المبيعات. |
| **العميل/المورد (Customer/Supplier)** | بناءً على العميل أو المورد. | عملاء VIP يحتاجون موافقة خاصة. |
| **المنتج/الخدمة (Product/Service)** | بناءً على المنتج أو الخدمة. | منتجات معينة تحتاج موافقة فنية. |
| **القسم (Department)** | بناءً على القسم. | طلبات قسم المشتريات تحتاج موافقة مدير المشتريات. |
| **الموقع (Location)** | بناءً على الموقع أو الفرع. | فواتير فرع معين تحتاج موافقة مدير الفرع. |
| **التاريخ (Date)** | بناءً على التاريخ أو الفترة الزمنية. | طلبات في نهاية الشهر تحتاج موافقة إضافية. |

### 4.3 تجاوز الموافقة (Approval Override)
- **الغرض:** السماح بتجاوز قواعد الموافقة في حالات استثنائية.
- **الشروط:**
    - يجب أن يكون لدى المستخدم صلاحية التجاوز.
    - يجب تسجيل سبب التجاوز.
    - يجب توثيق التجاوز في سجل التدقيق.
    - قد يتطلب التجاوز موافقة إضافية من مدير أعلى.

## 5. إدارة المهام (Task Management)

### 5.1 أنواع المهام
| النوع | الوصف |
| :--- | :--- |
| **مهمة موافقة (Approval Task)** | مهمة تتطلب موافقة أو رفض. |
| **مهمة مراجعة (Review Task)** | مهمة تتطلب مراجعة وتقديم ملاحظات. |
| **مهمة تنفيذ (Execution Task)** | مهمة تتطلب تنفيذ إجراء معين. |
| **مهمة إعلامية (Notification Task)** | مهمة لإعلام المستخدم بمعلومة معينة. |

### 5.2 بيانات المهمة
| الحقل | الوصف | إلزامي |
| :--- | :--- | :--- |
| **رقم المهمة (Task ID)** | رقم فريد للمهمة. | نعم |
| **الاسم (Name)** | اسم المهمة. | نعم |
| **النوع (Type)** | موافقة، مراجعة، تنفيذ، إعلامية. | نعم |
| **المستخدم المسؤول (Assignee)** | المستخدم المسؤول عن المهمة. | نعم |
| **المستخدم المنشئ (Creator)** | المستخدم الذي أنشأ المهمة. | نعم |
| **المعاملة المرتبطة (Related Transaction)** | المعاملة المرتبطة بالمهمة. | نعم |
| **الحالة (Status)** | معلقة، قيد التنفيذ، مكتملة، ملغاة. | نعم |
| **الأولوية (Priority)** | عالية، متوسطة، منخفضة. | نعم |
| **تاريخ الإنشاء (Created Date)** | تاريخ إنشاء المهمة. | نعم |
| **تاريخ الانتهاء المطلوب (Due Date)** | التاريخ المطلوب لإنهاء المهمة. | لا |
| **تاريخ الإنجاز (Completed Date)** | تاريخ إنجاز المهمة. | لا |
| **الملاحظات (Notes)** | أي ملاحظات إضافية. | لا |
| **المستندات المرفقة (Attachments)** | المستندات المرفقة بالمهمة. | لا |

### 5.3 قواعد المهام
- **التوزيع التلقائي:** توزيع المهام تلقائياً على المستخدمين بناءً على قواعد سير العمل.
- **التذكيرات:** إرسال تذكيرات تلقائية للمستخدمين عند اقتراب موعد انتهاء المهمة.
- **التصعيد:** تصعيد المهام غير المنجزة إلى مدير أعلى بعد فترة محددة.
- **التفويض:** إمكانية تفويض المهمة إلى مستخدم آخر.
- **الإلغاء:** إمكانية إلغاء المهمة (مع تسجيل السبب).

## 6. تتبع الحالة (Status Tracking)

### 6.1 حالات المعاملة الأساسية
| الحالة | الوصف |
| :--- | :--- |
| **مسودة (Draft)** | المعاملة في مرحلة الإنشاء (لم ترسل بعد). |
| **قيد المراجعة (Under Review)** | المعاملة قيد المراجعة من قبل المدققين. |
| **معلقة (Pending)** | المعاملة في انتظار إجراء من مستخدم آخر. |
| **مرفوضة (Rejected)** | تم رفض المعاملة من قبل أحد المدققين. |
| **معتمدة (Approved)** | تمت الموافقة على المعاملة. |
| **منتهية (Completed)** | اكتملت جميع خطوات المعاملة. |
| **ملغاة (Cancelled)** | تم إلغاء المعاملة (بقرار من المستخدم أو النظام). |

### 6.2 سجل الحالة (Status History)
| الحقل | الوصف |
| :--- | :--- |
| **التاريخ (Date)** | تاريخ التغيير. |
| **الحالة السابقة (Previous Status)** | الحالة قبل التغيير. |
| **الحالة الجديدة (New Status)** | الحالة بعد التغيير. |
| **المستخدم (User)** | المستخدم الذي قام بالتغيير. |
| **السبب (Reason)** | سبب التغيير (إن وجد). |
| **الملاحظات (Notes)** | أي ملاحظات إضافية. |

## 7. الإشعارات المرتبطة بسير العمل (Workflow Notifications)

### 7.1 أنواع الإشعارات
| النوع | الوصف |
| :--- | :--- |
| **إشعار مهمة جديدة (New Task)** | إعلام المستخدم بوجود مهمة جديدة عليه. |
| **إشعار تذكير (Reminder)** | تذكير المستخدم بمهمة معلقة. |
| **إشعار تصعيد (Escalation)** | إعلام المستخدم بتصعيد مهمة إلى مستوى أعلى. |
| **إشعار تغيير الحالة (Status Change)** | إعلام المستخدمين بتغيير حالة المعاملة. |
| **إشعار موافقة (Approval)** | إعلام المستخدم بالموافقة على طلبه. |
| **إشعار رفض (Rejection)** | إعلام المستخدم برفض طلبه (مع ذكر السبب). |
| **إشعار إكمال (Completion)** | إعلام المستخدم باكتمال سير العمل بالكامل. |

### 7.2 قواعد الإشعارات
- **التوقيت:** إرسال الإشعارات في الوقت المناسب (عند إنشاء المهمة، عند التذكير، عند التغيير).
- **القناة:** إرسال الإشعارات عبر القناة المفضلة للمستخدم (بريد إلكتروني، Push، داخل التطبيق).
- **التخصيص:** تخصيص محتوى الإشعار بناءً على نوع المعاملة والمستخدم.
- **التكامل:** التكامل مع وحدة الإشعارات لإرسال الإشعارات.

## 8. لوحات المعلومات والتقارير (Dashboards & Reports)

### 8.1 لوحات المعلومات (Dashboards)
- **لوحة المهام (Task Dashboard):** عرض المهام المعلقة، القادمة، المنتهية لكل مستخدم.
- **لوحة الموافقات (Approval Dashboard):** عرض طلبات الموافقة المعلقة، المعتمدة، المرفوضة.
- **لوحة أداء سير العمل (Workflow Performance Dashboard):** عرض متوسط وقت المعالجة، عدد المعاملات، نسب الموافقة/الرفض.

### 8.2 التقارير (Reports)
| التقرير | الغرض |
| :--- | :--- |
| **تقرير أداء سير العمل (Workflow Performance)** | عرض أداء سير العمل (متوسط الوقت، عدد المعاملات). |
| **تقرير الموافقات (Approval Report)** | عرض جميع طلبات الموافقة وحالاتها. |
| **تقرير المهام (Task Report)** | عرض جميع المهام وحالاتها. |
| **تقرير التصعيدات (Escalation Report)** | عرض المهام التي تم تصعيدها. |
| **تقرير الانتهاكات (Violation Report)** | عرض حالات تجاوز المهلة أو انتهاك القواعد. |

### 8.3 مؤشرات الأداء (KPIs)
| المؤشر | الوصف |
| :--- | :--- |
| **متوسط وقت المعالجة (Average Processing Time)** | متوسط الوقت من إنشاء المعاملة حتى اكتمالها. |
| **نسبة الموافقة (Approval Rate)** | نسبة المعاملات المعتمدة إلى إجمالي المعاملات. |
| **نسبة الرفض (Rejection Rate)** | نسبة المعاملات المرفوضة إلى إجمالي المعاملات. |
| **معدل الالتزام بالمهلة (SLA Compliance Rate)** | نسبة المعاملات المنجزة ضمن المهلة المحددة. |
| **متوسط وقت الموافقة (Average Approval Time)** | متوسط الوقت من طلب الموافقة حتى الرد. |

## 9. العمل في وضع الأوفلاين (Offline Mode)

### 9.1 السلوك في وضع الأوفلاين
- **إنشاء الطلبات:** يمكن إنشاء طلبات جديدة في وضع الأوفلاين (تنتظر المزامنة).
- **الموافقات:** لا يمكن تنفيذ الموافقات في وضع الأوفلاين (تنتظر الاتصال).
- **عرض المهام:** عرض المهام المخزنة محلياً (قد تكون غير محدثة).
- **تحديث الحالة:** تحديث حالة المهام محلياً (تنتظر المزامنة).

### 9.2 المزامنة (Synchronization)
- **رفع البيانات:** رفع الطلبات الجديدة وتحديثات الحالة.
- **تنزيل البيانات:** تنزيل تحديثات المهام والموافقات من الخادم المركزي.
- **سياسة النزاعات:** Server-Wins لجميع عمليات سير العمل (الخادم هو المصدر الأساسي).

## 10. التوسعات المستقبلية (Future Extensions)

- **الذكاء الاصطناعي (AI):** التنبؤ بوقت المعالجة، اقتراح المدققين المناسبين، اكتشاف الحالات الشاذة.
- **التعلم الآلي (ML):** تحسين سير العمل بناءً على البيانات التاريخية.
- **سير العمل الديناميكي:** تغيير مسار سير العمل تلقائياً بناءً على سلوك المستخدمين والظروف المتغيرة.
- **التكامل مع أنظمة خارجية:** ربط سير العمل مع أنظمة خارجية (مثل أنظمة إدارة العقود، أنظمة الموارد البشرية الخارجية).
- **الموافقة الصوتية:** دعم الموافقة عبر الأوامر الصوتية (للمدراء في الاجتماعات).

---

## 11. متطلبات الأداء (Performance Requirements)

- **معالجة طلب:** أقل من ثانية.
- **إنشاء مهمة:** أقل من 500 مللي ثانية.
- **إرسال إشعار:** أقل من ثانية.
- **توليد تقرير:** أقل من 3 ثوانٍ.
- **معالجة 10,000 طلب في الساعة:** استخدام قوائم الانتظار والمعالجة الدفعية.

## 12. سيناريوهات الإجهاد المؤسسية (Enterprise Stress Scenarios)

| السيناريو | آلية المواجهة |
| :--- | :--- |
| **آلاف الطلبات في اليوم** | استخدام قوائم الانتظار (Queues)، المعالجة الدفعية (Batch Processing). |
| **مواسم الذروة (مثل نهاية السنة المالية)** | زيادة سعة الخوادم، تحسين أداء الاستعلامات. |
| **فشل خدمة الإشعارات** | التبديل إلى قناة احتياطية، إعادة المحاولة التلقائية. |
| **تغيير مفاجئ في قواعد الموافقة** | تحديث تعريفات سير العمل بسرعة، تنبيه المستخدمين بالتغيير. |
| **فشل الخادم المركزي** | استخدام النسخ الاحتياطي (Replicas) وخطط التعافي من الكوارث. |

**القرار المؤسسي:**  
تم اعتماد هذه المواصفات لتكون الوثيقة المرجعية النهائية (Single Source of Truth) لمحرك سير العمل والموافقات في نظام Smart Merchant ERP. لا يُسمح للمطورين بابتكار عمليات خارج نطاق هذا المستند، وأي تحديث يتطلب مراجعة وإقراراً من فريق الهندسة المعمارية (Architecture Team).
